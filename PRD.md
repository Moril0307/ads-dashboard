1. 项目目标与背景
简述为什么要写这个后台。

内容：聚合 6 个 Google Ads 账户数据，并与手动上传的 CSV 服务端数据对齐，只关注“当前仍在投放或近期投放的、状态为开启的 campaign”的数据。

痛点：解决 Ads 后台转化数据与真实付费数据不一致的问题，自动计算真实 CPA，并能快速发现“在烧钱但没人付费”的系列。

2. 核心功能需求（最重要）
详细描述 AI 需要实现的功能逻辑。

**数据来源（简化约定）**
- **日期、campaign、预算、消耗**：来自你上传的**第一张 CSV（指标表）**。表头支持中文：**日期**（或 天）、**广告系列**、**预算**、**费用**。系统按「日期 + 广告系列」匹配并更新到 `ads_metrics`。
- **新 JID 付费人数**：来自你上传的**第二张 CSV（新 JID 付费表）**。表头支持中文：**日期**、**广告系列**、**新 JID 付费数量**。系统按「日期 + 广告系列」与指标表匹配，匹配上的行参与 CPA 计算，对不上的行该列显示为 `-`。
- **新 IOS JID 数量（可选）**：同样来自**第二张 CSV（新 JID 付费表）**的扩展列。表头兼容：**新IOS JID数量**（也兼容 **新 JID 付费表** 这类导出列名）。系统写入 `server_paid_data.new_ios_jid_users` 并在透视表中展示；缺失或无匹配记录时显示为 `-`。
- **新用户 CPA**：**消耗 / 新 JID 付费人数**（当新 JID 付费人数为 0 或缺失时，CPA 显示为 `-`）。
- **新 IOS 用户 CPA**：**消耗 / 新 IOS JID 数量**（当新 IOS JID 数量为 0 或缺失时，CPA 显示为 `-`）。

2.1 指标表 CSV 上传
- 你上传一张 CSV，提供：日期（列名可为「日期」或「天」）、广告系列、预算、费用。
- 系统校验：日期格式 YYYY-MM-DD（印度时区自然日）、广告系列非空、预算与费用为非负数字；同一文件中「日期 + 广告系列」不可重复。
- 通过校验后，按「日期 + 广告系列」upsert 到 `ads_metrics`，并自动根据 campaign 前缀填充产品线（ft / pu / ppt / other）。

2.2 新 JID 付费表 CSV 上传
- 你上传另一张 CSV，提供：日期、广告系列、新 JID 付费数量；可选提供：新 IOS JID 数量（表头兼容 **新IOS JID数量** / **新 JID 付费表** / `new_ios_jid_users`）。
- 系统按「日期 + 广告系列」与指标表（ads_metrics）匹配；匹配上的行写入 `server_paid_data`（`new_jid_users`，以及可选的 `new_ios_jid_users`），用于看板展示与 CPA 计算。
- 新用户 CPA = 消耗 / 新 JID 付费人数；当新 JID 付费人数为 0 或该「日期 + 广告系列」在付费表中无记录时，CPA 显示为 `-`。
- 新 IOS 用户 CPA = 消耗 / 新 IOS JID 数量；当新 IOS JID 数量为 0 或该列缺失/无记录时，CPA 显示为 `-`。

2.3 产品线自动分类
- 以 Ads 的 `campaign_name` 为唯一标准进行前缀识别和分类。
- 前缀规则（不区分大小写），前缀后面必须紧跟下划线 `_`：
  - Fachat：前缀 `ft_`
  - Parau：前缀 `pu_`
  - Pinkpinkchat：前缀 `ppt_`
- 不满足上述三个前缀规则的 campaign 自动归类为 `Other` 产品线，但在当前版本中顶部汇总卡片只展示 `ft` / `pu` / `ppt` 三条，不做 `Other` 的顶部汇总卡片。

2.4 统一时区与日期范围
- 全系统所有日期一律以 GMT+05:30 (Asia/Kolkata，印度时间) 为标准：
  - Google Ads 账户本身也使用该时区。
  - CSV 中的日期也以此时区为准。
- 日期维度只精确到“天”，不需要小时级别的曲线。
- 当前端选择日期范围（例如 `2026-02-01 ~ 2026-02-07`）时：
  - 采用“闭区间”逻辑：包含起始日期和结束日期在内的所有天（共 7 天）。

2.5 每日任务调度
- 每天印度时间约 05:00 运行一轮定时任务：
  - 拉取截至“昨天”完整一天的 Ads 数据。
  - 接收/导入用户上传的 CSV，并对齐到相同日期范围。
- 目标是让用户在北京时间约 07:30 看到“截止昨天晚上”的完整数据。

3. 数据计算与校验逻辑
告诉 AI 公式和边界条件，避免随意发挥。

3.1 CSV 校验与导入规则
- 上传的 CSV 文件支持逗号、制表符或分号分隔，系统会自动识别真正的表头行（包含“日期/天、广告系列、预算、费用”或“日期、广告系列、新 JID 付费数量”）。
- 行级校验规则：
  - `date` 不能为空，且格式必须是 `YYYY-MM-DD`。否则视为该行无效。
  - `campaign_name` 不能为空。否则视为该行无效。
  - 指标表中的 `budget` 和 `spend` 必须是非负数字；新 JID 付费表中的 `new_jid_users` 必须是非负整数。否则视为该行无效。
  - 同一张表内，`date + campaign_name` 不能重复；若重复，后续重复行视为无效行。
- 导入策略：
  - 对于**有效行**，按 `date + campaign_name` 进行 upsert 写入数据库。
  - 对于**无效行**（包括日期格式错误、字段为空、主键重复等），跳过不写入，但会记录「第几行、错误原因」。
  - 当整张表**没有任何有效行**时，返回错误（400），提示该 CSV 完全无效。
  - 当存在至少一行有效数据时，接口返回部分成功结果：`插入行数`、`跳过行数` 以及被跳过行的行号和错误信息，前端在上传结果中展示：“成功导入 X 行，跳过 Y 行（第 xx, yy 行）”。

3.2 Ads 与 CSV 对齐键
- 核心关联键：`date + campaign_name`，两边均以 GMT+05:30 的自然日为准。
- 多账号情况：
  - 多个 Google Ads 账号之间的 `campaign_name` 保证不重名，因此无需额外 account_id 参与对齐。
  - 若未来出现重名，再扩展键为 `account_id + campaign_name + date`。

3.3 CPA 计算（单行）
- 真实 CPA 使用“新 JID 付费数量”字段计算：
  - 单行 CPA = `spend / new_jid_users`。
- 特殊情况处理：
  - 如果该行有 Ads 消耗（`spend > 0`），但：
    - CSV 中找不到对应的 `date + campaign_name` 记录，或
    - 找到了记录但 `new_jid_users = 0`，
    则：
    - `paid_users` 字段在表格中展示为 `-`（不显示 0，避免误解为“明确为 0 人付费”）。
    - CPA 字段展示为 `-`，避免除以 0。
  - 如果某一天完全没有 Ads 消耗（`spend = 0`）：
    - 在选定日期范围内，为了方便发现缺失数据，也可以显示一行，`spend = 0`，`paid_users = -`，`CPA = -`，前提是该 campaign 处于开启状态。

3.4 日期范围内的汇总 CPA
- 当用户选择一个日期范围（例如最近 7 天），对某个粒度（产品线、单条 campaign）进行汇总时：
  - 总消耗（Total Spend）= 该范围内所有天的 `spend` 之和（包含 `new_jid_users = 0` 或缺失的天）。
  - 总付费人数（Total Paid Users）= 该范围内所有天的 `new_jid_users` 之和。
  - 平均 CPA（Average CPA）按下式计算：
    - 如果总付费人数 > 0：`Average CPA = Total Spend / Total Paid Users`，并在界面上保留两位小数。
    - 如果总付费人数 = 0：`Average CPA` 显示为 `-`，但同时在 UI 上明确展示该产品线/该 campaign 的“总消耗”，让用户一眼看到“在烧钱但没人付费”。
- 对于单天 `paid_users` 显示为 `-` 的情况，在汇总时把该天视为 0 参与“总付费人数”的求和，但不单独显示为 0，避免视觉误导。

3.5 总览卡片与产品线维度
- 顶部展示 3 张汇总卡片，分别对应：
  - ft（Fachat）
  - pu（Parau）
  - ppt（Pinkpinkchat）
- 每张卡片展示选定日期范围内该产品线的：
  - 总消耗（总 spend）
  - 总真实付费人数（总 new_jid_users）
  - 平均 CPA（保留两位小数，若总付费人数为 0，则 CPA 显示为 `-`）
- 当前版本不做全局“总汇总”卡片，也不为 `Other` 产品线单独做卡片；若后续发现有需要，再扩展。

4. 交互与 UI 规范

4.1 看板视图
- 使用紧凑型 DataTable，支持以下筛选和排序（旧版列表视图，保留作对比参考）：
  - 日期范围选择（默认最近 7 天，闭区间）。
  - 产品线筛选（ft / pu / ppt / Other）。
  - campaign_name 搜索。
- 当前实现已升级为**产品透视表 + 日汇总列**，具体见下方补充说明。
- 排序默认规则：
  - 先按日期降序（最近日期在上）。
  - 再按产品线顺序：ft → pu → ppt → Other。
  - 再按 `campaign_name` 字母排序。
- 支持分页显示，默认每页 50 行，以避免大范围查询时前端卡顿。

4.1 补充：产品透视表 + 产品日汇总列
- 视图形态：采用「左侧日期 Sticky + 右侧多列 Campaign Group」的产品透视表布局，整体风格为 Excel 办公风。
- 顶部筛选与导航：
  - 日期范围选择（默认最近 7 天，闭区间）。
  - 产品线页签（Tabs）：Fachat / Parau / Pinkpinkchat，对应 ft / pu / ppt 三条产品线；切换 Tab 时仅加载对应产品线的数据。
- 纵轴（Y 轴）：
  - 第一列为日期列（印度时间自然日，例如 `2026年2月26日 星期四`），固定在左侧，在横向滚动时保持可见。
- 横轴（X 轴）结构：
  - 日期右侧首先是一组**产品日汇总列**（仅针对当前 Tab 所选产品线）：
    - 总预算 (USD)：汇总该产品线在该日期下所有广告系列的预算之和（在后端以 HKD 累加，在前端统一使用固定汇率 1 USD = 7 HKD 的 `toUsd()` 转换为 USD，保留两位小数）。
    - 总花费 (USD)：汇总该产品线在该日期下所有广告系列的消耗之和（同样在 HKD 空间累加，再用统一的 `toUsd()` 转为 USD，保留两位小数）。
    - 总 JID 付费数量：汇总该产品线在该日期下所有广告系列的 `new_jid_users` 之和（仅累加 `new_jid_users > 0` 的行）。
    - 总体 CPA (USD)：若总 JID 付费数量 > 0，则按 `总体 CPA = 总花费 / 总 JID 付费数量` 在 HKD 里先做除法，再用同一个 `toUsd()` 转为 USD，保留两位小数；若总人数 = 0，则显示为 `-`。
    - 当某天该产品线完全没有消耗和付费（总花费 = 0，总人数 = 0）时，这三列显示为 `0.00 / 0 / -`（总预算为 0.00，总花费为 0.00，总人数为 0，总体 CPA 为 `-`）。
  - 日汇总列右侧通过一列固定宽度约 40px 的**透明间隔列**与具体 Campaign 明细区隔开，形成清晰的视觉断层。
  - 具体 Campaign 区域：每个 `campaign_name` 作为一个列组横向排布，每个列组包含：
    - 备注（`date + campaign_name` 粒度，单元格内编辑、覆盖保存）
    - 预算（当日预算，原始为 HKD，在前端统一换算为 USD、保留两位小数）
    - 消耗（Spend，当日消耗，原始为 HKD，在前端统一换算为 USD、保留两位小数）
    - 付费（Ads 报告的转化数，可作为参考字段）
    - 新 JID 用户数（`new_jid_users`）
    - 新 JID CPA（行级 CPA = 当天消耗 / 新 JID 用户数，保留两位小数或显示为 `-`）
    - 新 IOS JID 数量（`new_ios_jid_users`，来自付费表扩展列；缺失或无匹配记录时显示为 `-`）
    - 新 IOS 用户 CPA（行级 CPA = 当天消耗 / 新 IOS JID 数量，保留两位小数或显示为 `-`）
  - 各个 Campaign 列组之间同样通过 40px 宽的透明列隔开，以强化系列间的视觉分组。
  - 日期维度在透视表中按**升序**展示（最早日期在上，最新日期在下），方便从过去到现在顺序阅读。
  - 为了减轻长表阅读压力，在从「周日」跨到「下周一」的位置自动插入一行**空白分割行**：
    - 不包含任何数据，`colSpan` 覆盖整张表，背景为纯白、不绘制边框，高度与普通数据行大致一致，用于清晰分隔不同周的数据块。
  - 在选定范围内的最后一天数据行下方，同样保留一行空白分割行，避免用户在滚动至底部或选择最后一行时被浏览器底部遮挡。

4.2 备注系统
- 分为两类备注：
  - daily_notes（“全局日记”）：
    - 一天内可以写多条，按创建时间排序。
    - 以日期（印度时间自然日）为维度，不绑定具体 campaign。
  - campaign_notes（“单条系列日记”）：
    - 粒度为 `date + campaign_name`。
    - 每个 `date + campaign_name` 只需要一条备注，可覆盖编辑，不需要记录历史版本；当用户在前端将备注内容清空并保存时，视为删除该条 `campaign_notes` 记录（数据库中对应行会被删除，界面上该单元格变为空白）。
    - **刷新时备注保留**：用户点击「刷新」后，前端会重新拉取当前视图下所有「日期 + 广告系列」的备注；若某条备注的拉取失败或未返回（如网络异常），则保留刷新前该单元格已显示的备注内容，不因整表替换而丢失。即：仅对当前表格中的每个 (date, campaign_name) 用接口结果更新，未拉取到的 key 沿用原 notesMap 中的值。
- “今天”的定义始终以印度时间为准：
  - 即使用户在其他时区操作，也按照印度时间日历来决定这条备注属于哪一天。
- 与筛选联动：
  - 当用户在看板中选择了日期范围 + 某条 campaign：
    - `campaign_notes` 区域只展示当前筛选出来的这些 `date + campaign_name` 对应的备注。
    - `daily_notes` 可以按日期范围过滤展示。

4.3 错误提示与防崩溃
- 无论任何错误（Ads 拉取失败、CSV 导入失败、数据库无数据），前端都不应白屏或崩溃：
  - 若选定范围内无数据，展示结构完整的“空表状态”，附带清晰的文案（例如“当前日期范围内没有可用数据”）。
  - 若 Ads 拉取失败：
    - 当天的数据视为“整体失败，不展示不完整数据”。
    - 顶部显示明显的红色提示条，说明“今天 Ads 数据未成功拉取，请检查后手动重试”。
    - 同时显示各个 Ads 账号的拉取状态（成功/失败），方便排查。
  - 若 CSV 导入失败：
    - 不写入任何新 CSV 数据。
    - 给出明确的错误原因（比如缺少列、重复键、某行字段非法等）。

5. 业务使用习惯补充

- 用户主要按“产品线 + campaign 维度 + 最近几天”查看数据，关注点集中在：
  - 总消耗（Total Spend）
  - 总真实付费人数（Total New JID Users）
  - 平均 CPA（Average CPA）
  - 以及总消耗与预算的对比（当总消耗超过预算时，认为该系列花费不稳定）。
- 不需要查看“历史已永久暂停且未来不再关注的 campaign”；系统实现时可以通过配置或过滤，聚焦在“当前仍有意义监控的系列”。
- 支持导出：
  - 原始上传的 CSV 备份。
  - 系统内部聚合后的明细数据（例如当前筛选条件下的 DataTable 导出），方便在 Excel 里做进一步自查。
